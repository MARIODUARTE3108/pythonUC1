O computador é obediente, não inteligente

Isso é o ponto mais importante de hoje, então vamos com calma: o computador não pensa. Ele é rápido e extremamente obediente — faz exatamente o que mandamos, nem mais, nem menos. Quando um programa dá errado (isso se chama bug), o problema quase sempre está na ordem que nós escrevemos, e não no computador em si.

☕ Vamos pensar juntos: pedir um café Se você disser para alguém "faça um café", sem mais detalhes, essa pessoa (ou um robô) pode travar — não sabe os passos exatos. Mas se você disser: "pegue a xícara → coloque uma colher de pó → ferva 200 ml de água → despeje", agora dá para seguir — e repetir isso 1000 vezes sem errar nunca. Programar é escrever ordens tão detalhadas quanto essa, sem deixar nada subentendido.

Algoritmo: a receita por trás de tudo

Um algoritmo é uma sequência finita e ordenada de passos para resolver um problema — pense numa receita de bolo:

  • A ordem importa: não asse antes de misturar.
  • Os passos não podem ser ambíguos: "bata 5 minutos", não "bata um pouco".
  • Tem que ter um fim garantido: a receita não pode rodar para sempre.

Guarde essa palavra: algoritmo. Vamos usá-la o curso inteiro.

Erros comuns — e por que eles acontecem

Errar faz parte. O importante é reconhecer o erro e saber corrigir. Clique para ver os dois erros mais comuns de quem está começando com print():

⚠️ Print("oi") com P maiúsculo ▾
Dá erro do tipo NameError. O Python diferencia maiúsculas de minúsculas (isso se chama case-sensitive): a função é sempre print, tudo minúsculo. Tente digitar Print("oi") no console acima para ver o erro de verdade — e depois corrija.
⚠️ print(oi) sem as aspas ▾
Sem aspas, o Python pensa que oi é o nome de uma variável (assunto da Aula 2!) — e como essa variável não existe, dá erro NameError. Texto sempre entre aspas.

⚠️ "Esqueci de definir a variável nos dois caminhos" ▾
Se uma variável só é criada dentro do if, e você tenta usá-la depois (fora do if/else) num caso em que o else rodou, dá NameError — a variável nunca existiu nesse caminho. Sempre garanta que toda variável usada depois do bloco condicional seja definida em todos os caminhos possíveis (vimos isso na Estação 7, com a variável total).
⭐ Curiosidade elif é uma palavra exclusiva do Python — outras linguagens usam "else if". E valores como 0, "" (texto vazio) e None contam como "falso" em uma condição, mesmo sem serem True/False explicitamente.

Local × Remoto: e se o seu PC quebrar hoje à noite?

🗂️ Repositório local × repositório remoto Até a Aula 3, seus commits ficavam só na sua máquina (o repositório local — pense nele como seu caderno pessoal). O repositório remoto é uma cópia na nuvem, no GitHub — pense nele como o Drive compartilhado do time.

Por que isso importa: backup (se o PC quebrar, o código continua seguro), colaboração (várias pessoas trabalham sem sobrescrever o trabalho uma da outra) e portfólio — seu GitHub é basicamente um currículo vivo que recrutadores olham.

Dicas de mercado e curiosidades

💼 No mercado push / pull / clone são o dia a dia de qualquer equipe de desenvolvimento. Recrutadores olham o seu GitHub — ter repositórios organizados, com commits e um bom README.md, conta muito em processos seletivos.

  • Nunca versione senhas ou dados sensíveis.
  • Escreva um README.md explicando o que o projeto faz.
  • pull antes, push depois — sempre.
🧪 Curiosidades O GitHub foi comprado pela Microsoft em 2018 por US$ 7,5 bilhões. origin é só um apelido que o Git dá para a URL do repositório remoto — você poderia chamá-lo de outro nome. Alternativas ao GitHub incluem GitLab e Bitbucket.
